针对你之前遇到的“登录态丢失”和“安全性”问题，我为你设计了一套业界目前普遍采用且更安全便捷的方案。

这套方案的核心思路是：**放弃在浏览器中直接存储Token，改用 `HttpOnly` Cookie 由服务端管理会话，并引入“短时效Access Token + 长时效Refresh Token”的双Token机制。**

这样做能同时解决你面临的三大痛点：

| 痛点 | 原方案问题 | 新方案解决方案 |
| :--- | :--- | :--- |
| **1. 新标签页需重登录** | `sessionStorage` 是标签页隔离的。 | `HttpOnly` Cookie 是同源共享的，天然支持多标签页。 |
| **2. XSS攻击风险** | `localStorage` 可直接被JS读取。 | `HttpOnly` Cookie 禁止JS访问，从根源免疫XSS窃取。 |
| **3. 会话时效管理** | 单一6小时长Token，风险窗口大。 | “短Token(内存)+长Token(Cookie)”模式，兼顾安全与体验。 |

具体有如下三种实现方案，你可以根据项目情况选择：

---

### 方案一：BFF (Backend for Frontend) 模式（最推荐）

这是目前架构上最优雅、安全性最高的方案。它的核心思想是增加一个 **BFF层** 作为前端和业务API之间的“代理”。

*   **核心流程**：
    1.  **登录**：前端将用户名密码发给BFF。
    2.  **存储**：BFF验证后，将JWT Access Token存于自己的内存或Redis中，并向前端浏览器下发一个仅用于标识会话的 `HttpOnly` Cookie。
    3.  **请求**：前端携带此Cookie请求BFF。
    4.  **转发**：BFF验证Cookie有效后，取出之前存储的JWT，附加到转发给业务API的请求头中。
*   **优势**：
    *   **极致安全**：前端的Cookie只用于标识“会话”，不包含任何业务Token信息。真正的JWT全程不接触浏览器，彻底杜绝XSS和CSRF风险。
    *   **解耦**：前端无需关心Token的存储和刷新逻辑，全部由BFF层处理。
*   **注意事项**：需要单独开发一个BFF服务，会增加一定的架构复杂度。

---

### 方案二：Access Token + Refresh Token 双Token机制（推荐）

如果你希望保持现有架构，这是最通用的选择。

*   **核心流程**：
    1.  **登录**：服务端登录成功后，返回两个Token：
        *   **Access Token**：**时效极短（如15分钟）**，用于调用业务API。
        *   **Refresh Token**：**时效较长（如7天）**，**仅**用于换取新的Access Token。
    2.  **存储**：
        *   **Access Token**：**存储在前端内存（Vuex/Pinia/Redux等状态管理中）**。页面刷新会丢失，但可通过Refresh Token无感恢复。
        *   **Refresh Token**：**存储在 `HttpOnly` 的Cookie中**。
    3.  **请求与刷新**：
        *   前端每次请求从内存中取出Access Token，放入`Authorization`头。
        *   当Access Token过期，API返回401，前端**自动**调用`/refresh`接口。该接口携带`HttpOnly` Cookie，服务端验证Refresh Token有效后，下发新的Access Token。整个过程对用户无感。
*   **优势**：
    *   **平衡了安全与体验**：核心业务Token（Access Token）存于内存，XSS无法窃取。Refresh Token虽在Cookie中，但因`HttpOnly`也无法被JS读取，且仅用于刷新，不直接暴露业务接口。
    *   **实现了“被动续期”**：只要Refresh Token未过期，用户就能一直无感使用。
*   **注意**：需要前端处理401状态码并发起刷新，实现稍复杂。

---

### 方案三：纯 HttpOnly Cookie + CSRF 防护（最简单）

如果你觉得双Token机制太复杂，这是一个快速提升安全性的方案。

*   **核心流程**：登录成功后，服务端直接将JWT放在 `HttpOnly` 的Cookie中返回。浏览器后续请求自动携带此Cookie。
*   **优势**：
    *   **实现简单**：无需前端额外存储逻辑，浏览器自动管理。
    *   **免疫XSS**：`HttpOnly` 属性防止JS读取。
*   **必须做**：**必须防范CSRF攻击**。因为Cookie会自动携带，所以需要增加防护，最简单的是使用 **SameSite=Strict/Lax** 属性。对于需要跨站携带的场景，则需实现 **CSRF Token** 机制。

---

### 总结与建议

这三种方案的安全性排序是：**方案一 (BFF) > 方案二 (双Token) > 方案三 (纯Cookie)**。

*   **追求极致安全与架构优雅**：选择 **方案一 (BFF)**。
*   **希望在现有架构上达到安全与体验的最佳平衡**：选择 **方案二 (双Token)**，这是目前最主流、推荐的做法。
*   **希望以最小改动快速上线**：选择 **方案三 (纯HttpOnly Cookie)**，但**务必**做好CSRF防护。

**最后，无论选择哪种方案，都请务必遵循以下安全基线**：

1.  **全站HTTPS**：防止Token在传输中被窃听。
2.  **设置安全的Cookie属性**：`HttpOnly`、`Secure`、`SameSite`。
